Sports VS + Draft Logo — Diagnosis & Fix Plan

Two reported issues, root-caused in code. One fix applied, one proposed. No engine/money logic touched.

BUG 1 Sports VS: no "VS", YES/NO don't show team names

Root cause

The card decides it's a VS question via isVsQuestion = sideLabels.yesLabel !== "YES" (QuestionCard.tsx:100). The side labels come from getQuestionSideLabels(), which uses either metadata.yes_label/no_label OR parses "TEAM1 vs TEAM2" from the title.

When an admin creates a VS match, the form collects Team A / Team B — but the create payload never sent them. The metadata block included only chart_symbol and reference_links — not yes_label/no_label (admin/page.tsx:1627). And the manual VS placeholder suggests "Who will win … Team A or Team B?" (uses "or", not "vs"), so the title-parse fallback also misses.

Result: labels fall back to plain "YES"/"NO" → isVsQuestion is false → no VS divider, no team names.

FIXED

The create payload now sends yes_label = Team A, no_label = Team B in metadata whenever VS mode is on. This makes the labels show team names and flips isVsQuestion true → the "vs" divider + two-logo layout render. Robust: works regardless of how the title is phrased.

Note: the "vs" divider also requires two logos to resolve (logos.length === 2). VS mode already collects two logo_keys, so that holds — provided the two logos exist in the library (see Bug 2).

BUG 2 Drafted questions: logos don't appear (Wikipedia image URLs)

Root cause

Two different logo systems that aren't bridged for drafts:

  • What the card reads: question.logo_keys → looked up in the logo library (/logos/active). The card does NOT read metadata.logo_url.
  • What draft import stores: the AI/JSON draft's logo_url / logo_url_b are saved into metadata.logo_url (main.py import_drafts ~6973) — a field the card never displays.

So a drafted question carries the Wikipedia URL in metadata.logo_url, but has no logo_keys → the card finds no logo → nothing shows. The supposed bridge, _backfill_question_logo_assets(), is currently a no-op stub (it only checks for existing keys; it never downloads the URL or creates a library asset).

Extra gotcha: the logo-download path rejects SVGs (main.py ~539, 563). Many Wikipedia logo links are .svg (e.g. club crests) → those fail even via the manual uploader. Your new example is a .jpg, which is fine. Use the "copy image address" of a .jpg/.png thumbnail, not the SVG.

Fix options (your call — proposed, NOT yet applied)

  • A. Bridge on import/approve (recommended): when a draft has metadata.logo_url, download it via the existing _download_logo_from_url, create a library asset, and attach its logo_key to the question. Then the card shows it like any other logo. Touches backend import/approve flow.
  • B. Card fallback: make the card render metadata.logo_url / logo_url_b directly as an <img> when no logo_keys resolve. Frontend-only, smaller change, but leaves the two logo systems separate.
  • C. Workflow only: don't auto-bridge — instead, in the admin draft-review UI, the admin picks library logos before publishing (the create flow already supports logo_keys). No code bridge; relies on the admin step.
Lowest-risk recommendation: Option B (card fallback to metadata.logo_url) — it's frontend-only, can't affect the engine, and immediately makes drafted Wikipedia .jpg/.png logos appear. Option A is "more correct" long-term but touches the backend import path.
